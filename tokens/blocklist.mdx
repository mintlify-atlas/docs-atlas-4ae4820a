---
title: 'ERC20 Tokens with Admin-Controlled Address Blocklists'
description: 'USDC and USDT let admins block specific addresses from transfers. A blocked contract address traps all funds it holds, creating serious counterparty risk.'
---

Some ERC20 tokens — most notably USDC and USDT — include an admin-controlled blocklist that prevents blocked addresses from participating in transfers. When an address is added to the blocklist, any call to `transferFrom` with that address as either source or destination will revert. This page covers how the pattern is implemented, which tokens use it, the risks it creates for protocols that hold or route these tokens, and practical defensive measures.

## Real-world examples

**USDC (Circle)** maintains a blocklist enforced by a `blacklister` role. Circle has blocked addresses in response to OFAC sanctions and law enforcement requests. As of 2024, hundreds of addresses have been blocked.

**USDT (Tether)** implements an `isBlackListed` mapping with equivalent semantics. Tether has exercised this capability multiple times in response to regulatory pressure and theft incidents.

## Source code

The following minimal implementation illustrates the pattern. Both `src` and `dst` are checked on every `transferFrom` call:

```solidity BlockList.sol
// Copyright (C) 2020 d-xo
// SPDX-License-Identifier: AGPL-3.0-only

pragma solidity >=0.6.12;

import {ERC20} from "./ERC20.sol";

contract BlockableToken is ERC20 {
    // --- Access Control ---
    address owner;
    modifier auth() { require(msg.sender == owner, "unauthorised"); _; }

    // --- BlockList ---
    mapping(address => bool) blocked;
    function block(address usr) auth public { blocked[usr] = true; }
    function allow(address usr) auth public { blocked[usr] = false; }

    // --- Init ---
    constructor(uint _totalSupply) ERC20(_totalSupply) public {
        owner = msg.sender;
    }

    // --- Token ---
    function transferFrom(address src, address dst, uint wad) override public returns (bool) {
        require(!blocked[src], "blocked");
        require(!blocked[dst], "blocked");
        return super.transferFrom(src, dst, wad);
    }
}
```

<Note>
  Notice that `transfer` is not overridden here — it delegates to `transferFrom` via the base `ERC20`, so the blocklist check applies to all transfer paths.
</Note>

## Integration risks

<Warning>
  A malicious or compromised token admin can block your contract's address at any time. When your contract address is on the blocklist, all token transfers involving it will revert — effectively trapping any funds the contract holds.
</Warning>

There are three plausible scenarios where a contract could become collateral damage:

**Regulatory action against the contract itself.** Regulators have directed token issuers to freeze funds held by specific protocols. A DEX pool, lending market, or bridge that holds USDC could be blocked if the protocol is deemed non-compliant in a jurisdiction where the token issuer operates.

**A single user is targeted.** If a sanctioned user has an open position in your protocol, the token issuer may choose to block your contract address rather than (or in addition to) the user's address directly. This harms all other users sharing that contract.

**Extortion.** A malicious actor in control of the token could threaten to block a high-value contract unless paid. The credible threat alone may be sufficient to extract value.

In all cases the impact is the same: your contract can no longer move the token, and any accounting logic that relies on being able to `transfer` or `transferFrom` will revert.

## Defensive measures

**Monitor blocklist events.** Subscribe to the `AddedBlacklist` / `RemovedBlacklist` events (USDT) or equivalent events on the tokens you integrate. Alert on any event where your contract address or a critical dependency is the subject.

**Consider a circuit-breaker on detection.** MakerDAO's TUSD adapter demonstrates a practical pattern: when an upgrade or anomalous condition is detected, the adapter freezes further interaction with the token and allows only withdrawals. You can apply the same idea to a blocklist: if your monitoring detects that your contract has been blocked, trigger a permissioned `cage()` or `pause()` that halts new deposits while still allowing existing holders to exit.

**Use wrapper contracts at system boundaries.** The readme for this project recommends isolating all direct ERC20 interaction into dedicated adapters at the edge of your system. This limits blast radius — only the adapter needs to handle the revert, and the core accounting remains clean.

**Maintain an allowlist.** For systems where it is practical, restrict which tokens can be deposited to a curated set of known-good tokens, and document the blocklist risk in your security disclosure.

<Tip>
  If you integrate USDC or USDT, consider what your protocol should do if the token becomes permanently un-transferable from your contract address. Having a documented recovery path before an incident occurs is significantly less costly than designing one under pressure.
</Tip>
